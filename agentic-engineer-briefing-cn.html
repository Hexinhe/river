📌 核心摘要

作者认为：不要追逐最新工具。基础 AI 公司会不断将有用的功能整合到产品中。你不需要安装各种包和工具，只需要理解几个基本原则：精准上下文、分离研究与应用、用规则和技能来管理偏好。

🎯 核心观点：不要追逐工具

你是一个开发者，每天用着 Claude 或 Codex CLI，心里却犯嘀咕：我真的把 AI 用透了吗？看到别人好像在造火箭，自己却连两块石头都堆不起来...

别急着买更多"装备"（插件、工具包、复杂的 CLAUDE.md）。作者说：你根本不需要那些！

🏢 基础公司在飞奔

如果一个问题真的有很好的解决方案，OpenAI、Anthropic 这些基础公司会是最大的用户，然后马上把这个方案整合到产品里。

🔄 每一代都在进化

新一代理 AI 会更愿意遵守指令。你需要不断重新思考什么是最优的——这就是为什么少即是多。

✨ 你不需要额外工具

技能、记忆 harness、子代理等都曾是"解决方案"，最终都被整合到基础产品中。相信这个过程！

📚 上下文就是一切

使用太多插件和外部依赖的最大问题是上下文膨胀——你的 Agent 被太多信息淹没了！

"你想给 Agent 精确到只提供完成它任务所需的信息，除此之外不要更多！"

❌ 错误示例

让 Agent 写一个 Python 猜谜游戏，它却要记得 26 个会话前的"内存管理"笔记...

✅ 正确做法：分离研究和实现

把研究（Research）和实现（Implementation）完全分开。让不同的 Agent 处理不同阶段，保持上下文干净。

⚡ 有效工作的原则

1. 实现要精准

❌ "去建一个认证系统"

→ Agent 需要自己研究什么是认证系统

✅ "用 JWT 实现认证，bcrypt-12 哈希，7天过期的刷新令牌"

→ Agent 精确知道要做什么

2. 先研究，后实现

如果你不确定具体实现方案：

① 先让一个 Agent 研究各种方案
② 然后让全新的 Agent 用干净的上下文来实现

3. 测试即终点

测试是非常好的里程碑——它们是确定性的，可以设置明确的期望。

"除非这 X 个测试通过，否则任务不算完成；而且你不准修改测试。"

🎭 谄媚陷阱（Sycopancy）

AI Agent 被设计成尽可能同意你、服从你。这导致了很有趣的现象：

"让它找 Bug，它会给你找 Bug——即使需要编造一个。"

因为它太想听你的话了！

解决方案：使用中性提示

❌ "帮我找数据库里的 Bug"

✅ "搜索数据库，跟随每个组件的逻辑，报告所有发现"

🚀 高级技巧：对抗性 Agent

① Bug 寻找 Agent：找出所有 Bug（会过度报告）

② 对抗 Agent：尝试反驳每个 Bug（会过度否认）

③ 裁判 Agent：综合两者，得出真相

💫 上下文膨胀问题

Agent 有时像地球上最聪明的存在，有时又蠢得可怜——区别就在于是否需要"填补空白"做假设。

压缩（Compaction）后的关键动作

每次压缩后，Agent 需要：

  • 📖 重新阅读任务计划
  • 📖 重新阅读相关文件
  • 🎯 继续工作前先确认

📋 Rules 和 Skills 的正确用法

Rules（规则）

如果你不想让 Agent 做某事，就写一条规则：

"写代码前先读 coding-rules.md"

"如果写测试，先读 test-rules.md"

"如果测试失败，读 failing-rules.md"

Skills（技能）

Skills 更适合编码做事的具体方法。如果你想让某事以特定方式完成，把它写成一个 Skill。

CLAUDE.md 应该是什么？

一个逻辑嵌套的目录，告诉 Agent 在什么场景下该读什么文件。保持尽可能简洁！

🤖 自动化与长时运行

不要让 Agent 永远运行

长时间运行会导致上下文膨胀。作者建议：每个 Contract 用一个新 Session。

Contract（契约）模式

创建 {TASK}_CONTRACT.md，里面明确指定：

  • ✅ 需要通过的测试
  • ✅ 需要验证的截图
  • ✅ 其他验收标准

设置 stop-hook，直到 Contract 完成才能结束会话。

🏁 总结：成功的公式

1️⃣ 保持简单

从最小配置开始，给基础 CLI 一个机会

2️⃣ 用 Rules 管理偏好

Agent 做错就加规则，告诉它不要再犯

3️⃣ 用 Skills 管理流程

把确定性的方法写成 Skills

4️⃣ 定期清理

规则和技能会相互矛盾，定期让 Agent 整理和去重

这就是秘诀。

保持简单，把 Rules 和 Skills 当作目录，

虔诚地注意它们的上下文和设计限制。

享受和未来的玩具一起工作的乐趣吧！🚀
